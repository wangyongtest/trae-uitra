## 一、技术栈最终选型

### 1.1 桌面框架决策：Electron（推荐）

虽然 Wails v3 和 Tauri 在性能上更优，但基于以下因素选择 **Electron**：

| 考量维度 | Electron | Tauri | Wails v3 |
|---------|---------|-------|---------|
| 团队经验 | ✅ 有Vue3+Electron项目经验 | ❌ 需学Rust | ⚠️ 需熟悉Wails绑定 |
| Go内核集成 | Sidecar子进程(stdio JSON-RPC) | Sidecar子进程 | ✅ 原生Go绑定 |
| 打包体积 | ~150MB | ~10MB+Go二进制 | ~15MB |
| 内存占用 | 200-400MB | 50-80MB | 60-80MB |
| 冷启动 | 2-4s | 1-2s | 0.5-1s |
| 生态成熟度 | ⭐⭐⭐⭐⭐ VS Code等验证 | ⭐⭐⭐⭐ 快速增长 | ⭐⭐ 小众 |
| 自动更新 | ✅ electron-updater成熟 | ⚠️ 需自行处理 | ⚠️ 基础支持 |
| IPC复杂度 | 中（3跳：Vue→preload→main→Go） | 中（Vue→Rust→Go） | 低（Vue→Go直连） |

**最终决策：Electron + Go Sidecar**
- 利用现有 Vue3+Electron 开发经验，降低学习成本
- Go 内核编译为独立二进制，通过 stdio JSON-RPC 与 Electron 主进程通信
- Sidecar 模式已有成熟实践（Claude Code、Continue 等均采用此模式）
- Phase 3 可评估迁移到 Tauri/Wails（架构上解耦，IPC 层抽象好即可）

### 2.4 精准缓存命中技术方案（Manus+Reasonix 融合）

基于 Manus 团队和 Reasonix 的实战经验，提升命中率的12条铁律：

```
┌────────────────────────────────────────────────────────────────┐
│               缓存命中精准化策略（12条铁律）                      │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  【铁律1】前缀锁定：System Prompt 零动态内容                     │
│  ──────────────────────────────────────────────               │
│  • 禁止在 system prompt 开头放时间戳（精确到秒=缓存全毁）        │
│  • 时间信息通过工具(get_time)获取，不写入前缀                    │
│  • request_id、session_id 等放 header/extra，不进消息体         │
│                                                                │
│  【铁律2】JSON 序列化确定性                                      │
│  ──────────────────────────────────────────────               │
│  • Tool Schema/Arguments 序列化必须固定 key 顺序                │
│  • Go 标准库 json.Marshal 对 map 无序 → 用 struct 或排序        │
│  • 浮点数、嵌套对象统一格式化，确保字节级一致                     │
│                                                                │
│  【铁律3】严格 Append-Only                                      │
│  ──────────────────────────────────────────────               │
│  • 消息历史只追加，永不修改/重排/删除已有消息                    │
│  • 工具并行结果按调用顺序排序，不按返回顺序                      │
│  • 压缩时只在"压缩点"替换，保持前面字节不变                     │
│                                                                │
│  【铁律4】动态内容隔离                                          │
│  ──────────────────────────────────────────────               │
│  • 工具返回的时间戳/随机ID/临时路径做脱敏处理                    │
│  • 例如 ls 输出的 inode/time 字段移除或固定化                   │
│  • bash 命令输出中动态路径替换为相对路径                        │
│                                                                │
│  【铁律5】工具遮蔽而非移除（Manus核心策略）                      │
│  ──────────────────────────────────────────────               │
│  • 不动态增删工具定义（会导致前缀缓存失效）                      │
│  • 通过 response_prefix 预填充约束可用工具                      │
│  • 例：用户刚输入时强制预填充 "<assistant>" 而非工具调用         │
│  • 例：编码阶段只允许 file_* 工具，通过前缀约束                 │
│                                                                │
│  【铁律6】压缩边界对齐                                          │
│  ──────────────────────────────────────────────               │
│  • 压缩点必须对齐在完整的 tool result 之后                      │
│  • 不产生孤儿 tool_call（有调用无结果）                         │
│  • 摘要替换区域从某个固定位置开始，前缀保持完整                  │
│  • 压缩是唯一可接受的"缓存重置点"，频率尽量低                   │
│                                                                │
│  【铁律7】模型切换 = Session切换                                │
│  ──────────────────────────────────────────────               │
│  • 不同模型维护独立 Session 和 KV 缓存                         │
│  • Planner 和 Executor 各自独立前缀（双模型协作）               │
│  • 切换模型时不尝试复用其他模型的缓存                           │
│                                                                │
│  【铁律8】多Session路由一致性                                   │
│  ──────────────────────────────────────────────               │
│  • 自部署vLLM时用Session ID路由到同一worker                    │
│  • API调用保持base_url和model参数稳定                          │
│  • 避免负载均衡导致同一Session打到不同后端                      │
│                                                                │
│  【铁律9】错误保留而非清除（Manus经验）                          │
│  ──────────────────────────────────────────────               │
│  • 失败的工具调用和错误堆栈保留在上下文中                       │
│  • 让模型从错误中学习，不隐式重试清除痕迹                      │
│  • 避免重试导致前缀变化（错误→重试→成功，字节变化）             │
│  • 策略：保留首次错误，后续重试用同格式添加                     │
│                                                                │
│  【铁律10】Tool命名前缀一致性                                   │
│  ──────────────────────────────────────────────               │
│  • 同类工具统一前缀：file_read / file_write / file_edit        │
│  • 支持前缀级别的工具遮蔽：预填充 "file_" 限制可用工具          │
│  • 不随意重命名工具（改名=schema变化=缓存失效）                 │
│                                                                │
│  【铁律11】Few-Shot多样性防惯性（Manus经验）                    │
│  ──────────────────────────────────────────────               │
│  • 示例模板引入微小随机变化（措辞/格式/顺序）                  │
│  • 防止模型陷入重复模式（连续读20个文件的惯性）                │
│  • 但变化幅度不能破坏前缀稳定性（前缀示例区保持固定）           │
│                                                                │
│  【铁律12】缓存命中率实时监控与自动诊断                         │
│  ──────────────────────────────────────────────               │
│  • 每轮请求后记录 prompt_tokens/cached_tokens/completion_tokens│
│  • 命中率<90%立即触发自动诊断（见监控面板设计）                │
│  • 诊断项：前缀变化/动态内容/压缩触发/Provider切换/工具变更    │
│  • 输出修复建议，严重时警告用户                                │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 三、上下文管理：从三层分区到五层架构

### 3.1 为什么三层分区是"基础版"

Reasonix 的三层分区（IMMUTABLE PREFIX / APPEND-ONLY LOG / VOLATILE SCRATCH）解决的是 **KV缓存命中** 问题，但没有解决：
1. **长程记忆问题**：100轮对话后上下文仍然爆炸，只能靠有损压缩
2. **工具数量爆炸问题**：MCP生态让工具从10个变成100个，污染前缀
3. **中间遗忘问题**：50+轮后模型忘记早期目标
4. **信息密度问题**：大量原始观测数据（网页/PDF/日志）挤占窗口
5. **多任务干扰问题**：并行子任务上下文互相干扰

### 3.3 五层架构关键数据流

```
用户新消息到达
    │
    ├──→ Layer 2 更新目标状态 (todo.md 改写=注意力操控)
    │
    ├──→ Layer 3 检索相关记忆 (BM25/codebase/文件系统)
    │    加载到 Layer 1 末尾（作为system-reminder或context块）
    │
    ├──→ [复杂任务?] Layer 4 派发子Agent
    │    子Agent独立Session处理 → 返回摘要 → 注入Layer 1
    │
    ├──→ Layer 0 (锁定) + Layer 1 (追加) + 检索片段 → 构造API请求
    │    注意：Layer 0 字节级完全一致 → 缓存命中核心
    │
    ├──→ API流式响应
    │    ├── 文本回复 → 追加到Layer 1
    │    ├── 工具调用 → 执行 → 脱敏结果追加到Layer 1
    │    └── NEEDS_PRO标记 → 切换到Pro模型(独立Session)
    │
    ├──→ Layer 1 超过阈值(窗口80%)?
    │    ├──→ Layer 4 派发压缩子Agent(Flash模型)
    │    ├──→ 在最近tool result边界对齐
    │    ├──→ 摘要替换历史内容（原始归档到文件系统Layer 3）
    │    └──→ 缓存重置点，但频率极低（低频压缩原则）
    │
    └──→ 更新Layer 2 (todo.md勾除完成项)
         更新命中率/成本统计
```

### 3.4 压缩策略：可恢复压缩（Manus原则）

传统压缩是有损的，压缩后信息永久丢失。可恢复压缩原则：

| 信息类型 | 压缩策略 | 恢复方式 |
|---------|---------|---------|
| 文件读取内容 | 替换为路径+行范围摘要 | read_file重新读取 |
| 网页/PDF内容 | 替换为URL+关键摘要 | 重新抓取（URL保留） |
| 终端输出 | 替换为命令+退出码+关键行 | bash重新执行（幂等命令） |
| 搜索结果 | 替换为查询+Top3结果摘要 | grep/glob重新搜索 |
| 对话历史 | 替换为结构化摘要 | 归档JSONL可回溯 |
| 错误堆栈 | 保留（不压缩错误信息） | - |

### 5.2 Go 内核 TDD 规范

**测试框架**：Go 标准库 `testing` + `testify/assert`（可选，保持零依赖优先）

**测试文件命名**：`xxx_test.go` 与源文件同目录

**测试命名规范**：
```go
// 格式：Test[模块]_[场景]_[预期结果]
func TestCacheMonitor_HitRate95Percent_ShowGreenStatus(t *testing.T) {}
func TestProvider_DeepSeekStream_TokenCountingAccurate(t *testing.T) {}
func TestCompact_AtToolResultBoundary_NoOrphanToolCall(t *testing.T) {}
func TestSerializer_JSONDeterministic_SameInputSameBytes(t *testing.T) {}
```

**TDD 红-绿-重构循环**：
```
1. RED：先写失败的测试，描述期望行为
2. GREEN：写最小代码使测试通过
3. REFACTOR：重构代码，保持测试通过
```

**每个模块的测试优先级**：

| 模块 | 优先级 | 关键测试点 |
|------|-------|-----------|
| `cache/monitor` | P0 | 命中率计算、Token计数、诊断规则、确定性序列化 |
| `provider/deepseek` | P0 | 流式调用、Token统计、缓存命中标记、错误处理 |
| `provider/volcengine` | P0 | 同上（火山特有字段适配） |

package cache

// 当命中率低于阈值时触发自动诊断。
//
// 线程安全：所有方法支持并发调用。
type Monitor struct {
    mu            sync.RWMutex
    totalPrompt   int64
    totalCached   int64
    totalComplete int64
    totalCost     float64
    history       []Snapshot
    diagnostics   []DiagnosticResult
}

// HitRate 返回当前会话的缓存命中率（0-1之间）。
//
// 计算公式：cachedTokens / promptTokens
// 无prompt token时返回1.0（避免除零）。
func (m *Monitor) HitRate() float64 {
    m.mu.RLock()
    defer m.mu.RUnlock()
    if m.totalPrompt == 0 {
        return 1.0
    }
    return float64(m.totalCached) / float64(m.totalPrompt)
}
