# TRAE Work × Reasonix 融合架构设计

> 产品代号：**TRAE Work Ultra**（暂定）
> 核心理念：以 TRAE Work 全场景平台为骨架，注入 Reasonix 极致缓存优化内核与命中率监测能力
> 目标：打造「场景全覆盖 + 成本极优化 + 体验最强化」的下一代 AI 原生工作台

---

│  • 重复造轮子，TRAE 已有能力无法复用                              │
│                                                                 │
│  策略C: 双内核共存（过渡方案 🔄）                                │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘

**推荐路径**：策略C（双内核过渡）→ 策略A（内核融合）
- Phase 1: 双内核并行，用户可切换，验证 Reasonix 内核在 TRAE 生态中的适配性
- Phase 2: 深度融合，Reasonix 内核成为 Code 模式默认引擎，命中率监测等能力反向赋能 Work 模式
- Phase 3: 统一内核，所有模式共享 Cache-First 架构优势

## 三、核心融合模块详细设计

### 3.1 Cache-First 上下文管理器（核心融合点）

**多模型缓存策略适配层**：Reasonix 的三层分区是针对 DeepSeek exact byte prefix 设计的。融合后需要为每个 Provider 实现缓存策略适配器：

```
┌─────────────────────────────────────────────────────────────────┐
│              Cache Strategy Adapter (缓存策略适配)               │
├──────────────────────┬──────────────────────────────────────────┤
│ Provider             │ 缓存策略                                 │
├──────────────────────┼──────────────────────────────────────────┤
│ DeepSeek (v3/v4)     │ • exact byte prefix (原Reasonix策略)      │
│                      │ • 三层分区严格执行                        │
│                      │ • 目标命中率 ≥99%                         │
├──────────────────────┼──────────────────────────────────────────┤
│ 豆包 (Doubao)        │ • prefix caching适配                     │
│                      │ • 缓存触发阈值对齐(如≥1024tokens)         │
│                      │ • 动态内容检测(时间戳/ID需隔离)            │
│                      │ • 目标命中率 ≥90%                         │
├──────────────────────┼──────────────────────────────────────────┤
│ OpenAI (GPT系列)     │ • prompt caching (层级缓存)               │
│                      │ • system/user/assistant边界对齐           │
│                      │ • 缓存断点标记                            │
│                      │ • 目标命中率 ≥85%                         │
├──────────────────────┼──────────────────────────────────────────┤
│ Anthropic (Claude)   │ • prompt caching (显式cache_control)      │
│                      │ • 支持多断点缓存                          │
│                      │ • 目标命中率 ≥85%                         │
├──────────────────────┼──────────────────────────────────────────┤
│ 自定义OpenAI兼容     │ • 保守策略(默认无缓存假设)                 │
│                      │ • 学习模式: 自动探测缓存行为               │
│                      │ • 用户可手动配置缓存策略                   │
│                      │ • 目标命中率 ≥70% (持续优化)              │
└──────────────────────┴──────────────────────────────────────────┘
```

**上下文分层规则（融合增强版）**：
- **IMMUTABLE PREFIX**：根据 Provider 缓存机制决定边界
  - 所有 Provider：system prompt、tool schema、project memory 初始化后锁定
  - 禁止在 prefix 中注入动态内容（时间戳、请求ID、状态标记）
  - 动态元数据放入请求 header 或 VOLATILE 区域
- **APPEND-ONLY LOG**：严格按序追加
  - 禁止上下文重排优化（attention 优化通过其他方式实现）
  - parallel-safe 工具并行执行后按调用顺序排序结果
  - 工具结果自动裁剪（超阈值压缩，但原始内容归档）
- **VOLATILE SCRATCH**：临时区域
  - R1 reasoning content 不发送给不支持的 Provider
  - Plan 状态、临时分析不污染 prefix
  - 仅蒸馏后的结论进入 LOG

#### 缓存失效自动诊断

当命中率低于阈值时，自动触发诊断：

| 诊断项 | 检测逻辑 | 修复建议 |
|-------|---------|---------|
| Prefix 污染 | 检查 system prompt 是否有动态内容（时间戳/ID） | 移除动态注入，改用 header 传递 |
| 消息重排 | 对比消息顺序哈希是否变化 | 关闭上下文重排优化 |
| 压缩触发 | 检查是否刚执行过 compaction | 正常现象，命中率会自动恢复 |
| 工具结果动态内容 | 扫描工具返回是否含时间戳/随机数/路径变化 | 对动态内容做脱敏/固定化处理 |
| Provider 切换 | 检查是否最近切换了模型 | 切换模型会重置缓存，属正常 |
| MCP 工具变更 | 检查 MCP 工具列表是否变化 | 工具 schema 变化会重置 prefix |
| 缓存过期 | Provider 端缓存 TTL 到期 | 正常现象，重新建立缓存 |
| 跨设备接续 | 从其他设备接续会话 | 正常现象，前缀可能有差异 |

## 五、关键技术决策与风险

### 5.1 技术决策记录

| 决策 | 选择 | 理由 | 替代方案 |
|------|------|------|---------|
| 桌面端技术栈 | Go Host + WebView | 性能+生态兼顾；复用现有 React 前端 | 纯 Wails（重写前端成本高）、Electron（性能差） |
| 内核语言 | Go (保留 Reasonix) | 性能优秀、单二进制、Reasonix 已有成熟代码 | Rust（学习曲线陡）、Node.js（性能差） |
| 前端框架 | 复用 TRAE 现有 React | 不重写 UI，降低成本 | 重写为原生（成本太高） |
| 缓存策略 | Per-Provider Adapter | 不同模型缓存机制差异大，需针对性优化 | 统一抽象（会损失深度优化能力） |
| 开源策略 | Phase 3 开源内核 | 先验证产品市场契合度，再开源建设生态 | 一开始就全开源（产品方向可能调整） |
| CLI 分发 | npm + brew + 单二进制 | 覆盖所有用户习惯 | 仅 npm（部分用户无 Node） |
| 双内核过渡 | Phase 1 保留原内核 | 降低风险，用户可选择，不强制切换 | 直接替换原内核（风险大） |

### 5.2 风险矩阵

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|---------|
| Reasonix 内核与 TRAE 前端集成复杂度高 | 高 | 高 | Phase 0 做 POC 验证；双内核过渡，保留原引擎 |
| 多模型缓存策略适配难度大 | 中 | 中 | 先适配最常用的 2 个模型，其他迭代；提供学习模式自动探测 |
| Desktop 混合架构性能不达标 | 中 | 中 | 早期做性能基准测试；必要时关键路径用 CGO/原生优化 |
| Reasonix 开源协议变化（MIT 改协议） | 低 | 高 | 内核做抽象层，预留替换空间；积极贡献上游 |
| 用户不接受新模式/习惯改变 | 中 | 中 | 极速模式默认关闭，用户主动开启；保留所有原有功能 |
| 命中率优化影响回答质量 | 低 | 高 | A/B 测试；保留上下文重排作为可选项；压缩质量评估 |
| Web 端无法运行 Go 内核 | 高 | 中 | Web 端使用 WASM 精简内核或云端代理；核心缓存逻辑在云端执行 |
